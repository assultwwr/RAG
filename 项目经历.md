# RAG知识库智能问答系统 - 项目经历

## 项目概述

**项目名称：** RAG知识库智能问答系统  

**项目周期：** 2026.01 - 2026.02  

**技术架构：** LangChain + FastAPI + Streamlit + ChromaDB + Ollama 

**项目角色：** 全栈开发工程师（独立完成）

### 项目简介

基于RAG（检索增强生成）技术构建的企业级知识库智能问答系统，支持多格式文档上传、智能检索、精准问答、会话管理等功能。系统采用前后端分离架构，后端使用FastAPI提供RESTful API，前端使用Streamlit构建交互式界面，实现了从文档处理到智能问答的完整闭环。

## 核心功能模块

### 1. 智能问答引擎（RAG Core）

**技术实现：**
- 构建LangChain执行链（RunnableWithMessageHistory），支持多轮对话上下文记忆
- 实现混合检索策略：向量检索（余弦相似度）+ BM25关键词检索，权重各占50%
- 集成CrossEncoder重排序模型（BGE-Reranker），对初筛结果进行二次精排，Top-K=5
- 设计动态检索器切换机制：无文档时仅用向量检索，有文档时启用混合检索+重排序

**关键代码：**
```python
# 混合检索 + 重排序链路
ensemble_retriever = EnsembleRetriever(
    retrievers=[vector_retriever, BM25_retriever], 
    weights=[0.5, 0.5]
)
compressor = CrossEncoderReranker(model=reranker_model, top_n=5)
retriever = ContextualCompressionRetriever(
    base_compressor=compressor,
    base_retriever=ensemble_retriever
)
```

**性能优化：**
- 检索结果格式化时自动更新文件访问记录，支撑智能清理策略
- 向量库刷新机制（refresh_vector_store），确保新增/删除文档后实时生效

### 2. 知识库管理系统

**核心功能：**
- **多格式文档上传：** 支持PDF、Word、TXT等格式，通过工厂模式动态选择加载器
- **智能去重机制：** 基于MD5哈希值检测重复内容，避免冗余向量存储
- **文档分块策略：** 递归字符分割（chunk_size=500, chunk_overlap=50），支持中英文标点分隔符
- **向量入库：** ChromaDB持久化存储，metadata包含文件名、创建时间等信息

**技术亮点：**
```python
# MD5去重逻辑
def upload_by_str(self, data: str, filename):
    md5_hex = get_string_md5(data)
    if check_md5(md5_hex):
        return "数据已存在"  # 秒级去重判断
    
    knowledge_chunks = self.spliter.split_text(data)
    self.chroma.add_texts(texts=knowledge_chunks, metadatas=[...])
    save_md5(md5_hex)
```

**删除同步机制：**
- 删除文件时同步清理向量数据、MD5记录、数据库元数据，保证数据一致性

### 4. 智能空间管理（自动化运维）

**核心功能：**

- **自动清理策略：** 可配置文件保留期限（0-120天）、会话保留期限（0-120天）
- **启动时自动检查：** 应用启动时自动扫描并清理过期数据，释放存储空间
- **实时监控统计：** 文件总数、活跃文件数、会话总数、磁盘占用（ChromaDB + 会话历史）

**技术实现：**
```python
# 自动清理流程
def run_auto_cleanup(self):
    # 1. 扫描过期文件（基于last_access_time）
    expired_files = self.db_manager.get_expired_files(retention_days)
    
    # 2. 逐个删除：向量库 → 数据库记录 → MD5记录
    for file in expired_files:
        self.knowledge_service.delete_by_filename_with_md5(filename, file_md5)
        self.db_manager.delete_file(filename)
    
    # 3. 清理过期会话
    deleted_sessions = self.session_manager.cleanup_expired(retention_days)
    
    # 4. 刷新向量库连接
    rag_service.refresh_vector_store()
```

**日志记录：**
- 完整的清理日志输出（`data/logs/space_manager.log`）
- 记录删除数量、释放空间、执行时间等关键指标

## 技术难点与解决方案

### 难点1：混合检索精度优化

**问题描述：**
- 单纯向量检索在关键词匹配场景下召回率低
- 单纯BM25检索无法理解语义相似性
- 初筛结果中包含大量低相关性文档，影响LLM生成质量

**解决方案：**
1. **EnsembleRetriever混合检索：** 结合向量检索（语义理解）和BM25（关键词匹配），权重各50%，兼顾召回率和准确率
2. **CrossEncoder重排序：** 使用BGE-Reranker对初筛的10个文档进行精细打分，选取Top-5作为最终上下文
3. **动态阈值调整：** 根据实际测试效果，设置similarity_threshold=10，平衡检索数量和响应速度

**效果验证：**
- 检索准确率提升约40%（主观评估）
- 答案相关性显著改善，减少"幻觉"现象

---

### 难点2：文档去重与增量更新

**问题描述：**
- 用户上传相同内容文件时，系统会重复存储向量，浪费存储空间
- 传统基于文件名的去重无法识别内容相同但文件名不同的情况
- 修改后的文件如何判断是否需要重新入库

**解决方案：**
1. **基于内容的MD5去重：** 对文档全文计算MD5哈希值，而非依赖文件名
2. **MD5持久化存储：** 将已处理文档的MD5记录到`data/md5.text`文件
3. **上传前快速校验：** `check_md5()`函数O(1)时间复杂度判断是否已存在
4. **删除时同步清理：** 删除文件时同时移除MD5记录，避免"假阳性"误判

**边界处理：**
- 短文本（<500字符）不进行分块，直接作为一个整体入库
- 长文本使用RecursiveCharacterTextSplitter智能分块，保持语义完整性

---

### 难点3：会话历史管理与内存优化

**问题描述：**
- Streamlit每次用户交互都会重新运行整个脚本，会话状态易丢失
- 长期运行的应用中，会话历史文件数量激增，占用大量磁盘空间
- 如何在重启应用后恢复之前的对话上下文

**解决方案：**
1. **st.session_state持久化：** 利用Streamlit内置会话状态管理当前对话
2. **文件系统存储：** 将会话历史序列化到JSON文件，key为session_id
3. **元数据索引加速：** 单独维护`session_metadata.json`，避免遍历所有会话文件
4. **自动过期清理：** 基于last_activity_time判断，超过30天未活动的会话自动删除
5. **启动时自动检查：** 应用启动时执行一次清理任务，标记`cleanup_done`防止重复执行

**性能优化：**
- 会话文件命名规范化（替换`:``-`为`_`），避免文件系统兼容性问题
- 元数据只保存首条消息前100字符作为摘要，减少存储开销

---

### 难点4：向量库实时更新与一致性

**问题描述：**
- 上传/删除文档后，RagService中的vector_store_service仍指向旧数据
- 新上传的文档无法立即被检索到
- 删除的文档仍在检索结果中出现

**解决方案：**
1. **显式刷新机制：** 提供`refresh_vector_store()`方法，重新初始化VectorStoreService
2. **上传后自动刷新：** 文件上传接口调用`rag_service.refresh_vector_store()`
3. **删除后自动刷新：** 文件删除接口同样触发刷新
4. **清理后自动刷新：** 空间管理清理任务完成后刷新向量库

**代码实现：**
```python
def refresh_vector_store(self):
    """刷新向量数据库连接，确保获取最新数据"""
    self.vector_store_service = VectorStoreService(
        embedding=OllamaEmbeddings(model=config.embeddings_model_name)
    )
    self.chain = self.__get_chain()  # 重建执行链
```

### 难点5：Docker环境下的模型路径适配

**问题描述：**
- 本地开发环境和Docker容器环境的模型文件路径不同
- Rerank模型在本地使用GPU加速，Docker中只能使用CPU
- 硬编码路径导致跨环境部署失败

**解决方案：**
1. **环境变量检测：** 通过`os.path.exists()`判断当前运行环境
2. **条件路径配置：**
   ```python
   if os.path.exists("/app/models/BAAI-bge-reranker-v2-m3"):
       # Docker环境
       RERANKER_MODEL_PATH = "/app/models/BAAI-bge-reranker-v2-m3"
       RERANKER_DEVICE = "cpu"
   elif os.path.exists(r"D:\...\models\BAAI-bge-reranker-v2-m3"):
       # 本地环境
       RERANKER_MODEL_PATH = r"D:\...\models\BAAI-bge-reranker-v2-m3"
       RERANKER_DEVICE = "cuda"
   else:
       # 降级方案：从HuggingFace下载
       RERANKER_MODEL_PATH = "BAAI/bge-reranker-v2-m3"
       RERANKER_DEVICE = "cpu"
   ```
3. **Docker镜像预置模型：** 在Dockerfile中将模型文件COPY到容器内，避免运行时下载

### 难点6：多格式文档解析兼容性

**问题描述：**
- PDF文档存在表格、图片等非结构化内容，提取困难
- Word文档可能包含复杂格式（页眉页脚、文本框）
- 不同编码的TXT文件可能出现乱码

**解决方案：**

1. **工厂模式设计：** FileLoaderFactory根据文件扩展名动态选择加载器
2. **统一接口封装：** 所有加载器返回LangChain Document对象，后续处理逻辑一致
3. **编码容错处理：** TextLoader默认使用UTF-8编码，支持自定义encoding参数
4. **可扩展架构：** 提供`register_loader()`方法，支持动态注册新的文件类型

**支持的格式：**
- `.txt` → TextLoader（纯文本）
- `.pdf` → PyPDFLoader（PDF文档）
- `.docx` → Docx2txtLoader（Word文档）

### 难点7：大规模向量检索性能优化

**问题描述：**
- 随着文档数量增加，向量检索响应时间线性增长
- ChromaDB默认配置在小规模数据下表现良好，但千级文档量时出现延迟
- 混合检索需要同时查询向量库和BM25索引，双重开销

**解决方案：**
1. **HNSW索引优化：** ChromaDB底层使用HNSW算法，自动构建近似最近邻索引
2. **限制检索数量：** 设置similarity_threshold=10，避免返回过多无关文档
3. **重排序剪枝：** CrossEncoder只处理Top-10文档，降低计算复杂度
4. **定期重建索引：** 删除文档后调用`refresh_vector_store()`，触发索引优化

**未来优化方向：**
- 引入FAISS替代ChromaDB，支持GPU加速检索
- 实现分层检索策略：先粗筛再精排
- 添加缓存层：对高频查询结果进行缓存
